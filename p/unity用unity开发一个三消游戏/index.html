游戏简介

消消乐（Match-3）是一类广受欢迎的益智消除游戏，玩家通过交换、排列或点击等方式，将三个或以上相同元素连成一线进行消除，达成目标后通关。其核心玩法简单易上手，但策略性随关卡提升而增强。

核心玩法：

  • 玩家通过点击屏幕，从上方随机掉落不同种类的水果（如葡萄、橙子、苹果等）。
  • 相同水果碰撞后会合成更大的水果，例如两个葡萄合成一个樱桃，两个樱桃合成一个橙子，依此类推，最终目标是合成出最大的“西瓜”。
  • 游戏采用类似“俄罗斯方块 + 2048”的机制：水果受重力影响堆叠，若堆积超过屏幕顶部，则游戏结束

核心玩法机制

  1. 基本规则 在游戏棋盘（通常为网格状）中，通过交换相邻元素（上下或左右）使至少3个相同元素连成一行或一列，即可消除并得分。消除后上方或新生成的元素会填补空位，可能引发连锁反应。
  2. 目标类型
    • 积分关卡：在限定步数或时间内达到目标分数。
    • 任务关卡：消除特定元素（如冰块、障碍物）、收集道具或引导物品至底部等。
    • 生存模式：在无限关卡中挑战高分。
  3. 特殊元素与道具 消除更多元素或形成特殊组合会生成强化道具，增强策略性：
    • 直线消除（4个连成一行/列）：消除整行或整列。
    • 爆炸范围（5个连成“L”或“T”形）：消除周围3×3区域。
    • 全屏清道夫（5个连成一线）：消除棋盘上所有相同元素。
    • 组合效果：将特殊元素两两组合，触发大范围消除。
  4. 障碍设计 随着关卡推进，会加入多种障碍增加挑战：
    • 固定障碍：如岩石、链条，需多次相邻消除才能清除。
    • 动态障碍：如蔓延的藤蔓、下沉的淤泥，会限制操作空间。
    • 限制条件：步数限制、时间限制、不可移动的锁住元素等。

游戏/功能拆解

内容

背景图片、墙壁和地板、水果（运动中水果、待命水果）、死亡判定线、UI（分数）、音效（落地、合成）

逻辑

  1. 待命水果，鼠标按下时在x轴跟随鼠标，松开鼠标水果落下
  2. 水果落地时，水果之间互相碰撞、带弹性效果，相同的水果碰撞在一起会合成更大的水果
  3. 水果如果抵达了死亡判定线，游戏失败，重新加载场景
  4. 显示当前的分数

开发步骤

  1. 场景摆放
  2. 给水果和场景等游戏物体添加重力、碰撞、弹性等组件
  3. 将水果做成预制体
  4. 创建待命水果跟随鼠标移动
  5. 待命水果在鼠标松开后落下
  6. 相同水果合成
  7. 水果碰到/超出死亡线的逻辑
  8. 显示UI分数
  9. 音效

思路、问题及解决

1. 网格的制作思路

网格主要负责两样事情：

  • 管理一个一个可移动的piece/tile
  • 管理一个一个不可移动的cell

piecePrefab

用一个结构体来表示piecePrefab中需要存储的信息：piecePrefab的种类、piecePrefab的预制体。这两者又是有一一对应的关系，所以用一个字典来管理。我们会有很多个这样的piecePrefab，所以需要一个数组来放置这些piecePrefab的信息。

cell

直接遍历我们规定好大小的棋盘，然后挨个生成和，将它们作为Grid的子物体即可。

//=============== Grid.cs ===============
public class Grid : MonoBehaviour
{
    public enum EPieceType
    {
        Normal,
        Count
    }
    
    [Serializable]
    public struct PiecePrefab
    {
        public EPieceType type;
        public GameObject prefab;
    }
    
    [SerializeField] private int dimensionX;
    [SerializeField] private int dimensionY;
    
    private Dictionary<EPieceType, GameObject> _piecePrefabDist;
    
    [SerializeField] private PiecePrefab[] prefabs;

    [SerializeField] private GameObject cellPrefab;
    private void Start()
    {
        // 初始化piece prefab字典数组
        _piecePrefabDist = new Dictionary<EPieceType, GameObject>();
        for (int i = 0; i < prefabs.Length; i++)
        {
            if (!_piecePrefabDist.ContainsKey(prefabs[i].type))
            {
                _piecePrefabDist.Add(prefabs[i].type,prefabs[i].prefab);
            }
        }

        // 初始化格子prefab
        for (int row = 0; row < dimensionY; row++)
        {
            for (int col = 0; col < dimensionX; col++)
            {
                GameObject gridCell = Instantiate(cellPrefab, new Vector3(row, col, 0), Quaternion.identity);
                gridCell.transform.parent = transform;  // 把cell作为grid的子物体
            }
        }
    }
}

2. 使用GamePiece 类来代替 GameObject

这样处理的好处是：

  • 封装状态与行为：把行列、类型等数据和操作放在同一个类里，避免到处操作 GameObject
  • 类型安全与易维护：用 enum 管理类型，集中实现切换逻辑（SetType/Init），减少重复代码
  • 更容易初始化/更新：Init 或 SetType 可以在内部处理视觉、动画、碰撞等更新，调用方只需传入新类型
  • 性能与可测试：可以缓存组件引用（SpriteRenderer、Collider），便于单元测试与优化
  • 更好扩展性：以后要加掉落、交互、网络同步、序列化都在 GamePiece 内部处理更清晰

后面两条的好处暂时还没有体会，可能在后面的课程中会有体现。

//=========== GamePiece.cs ============
public class GamePiece : MonoBehaviour
{
    // 位置信息，使用属性，只需getter
    public int Row { get; private set; }
    public int Col { get; private set; }
    // type信息
    public Grid GridRef { get; private set; }
    public Grid.EPieceType EType { get; private set; }

    // Unity不支持有参构造函数，且生命周期函数也不能带参，所以用一个Init函数较合适
    public void Init(int row, int col, Grid grid, Grid.EPieceType type)
    {
        Row = row;
        Col = col;
        GridRef = grid;
        EType = type;
    }
}

//============ Grid.cs ============
 private void Start()

        _pieces = new GamePiece[dimensionX, dimensionY];
        for (int row = 0; row < dimensionY; row++)
        {
            for (int col = 0; col < dimensionX; col++)
            {
                GameObject newPiece = Instantiate(_piecePrefabDist[EPieceType.Normal], GetWorldPos(col,row), Quaternion.identity);
                newPiece.name = "Piece(" + row + "," + col + ")";
                newPiece.transform.parent = transform;

                _pieces[col, row] = newPiece.GetComponent<GamePiece>();
                _pieces[col,row].Init(row, col, this, EPieceType.Normal);
            }
        }
    }

3. 二维数组坐标与坐标轴x/y易搞混

针对类似的有网格的项目这种情况非常常见，是个非常容易搞错和混淆的大坑。可以在项目初期就定下统一的规则，后续全部按照这个规则来操作。

比如，在本项目中遵循的规则：

  • col 走 X轴（横向），范围：0..dimensionX-1
  • row 走 Y轴（纵向），范围：0..dimensionY-1
  • 数组用 _pieces[col, row]
  • 所有地方都严格按 (row, col) 传参、按 [col,row] 存取

4. 单一职责原则

使用 MovablePiece.cs 去控制piece的移动，用 ColorPiece.cs 来控制piece的渲染，避免到处操作 GameObject。

private int _row;
public int Row
{
    get
    {
        return _row;
    }
    set
    {
        _row = value;
    }
}

//========== 属性错误示例 ==========
// 如果写成这样，会造成无限递归
// 同时，属性应尽量保持轻量、无副作用；在属性里做复杂逻辑或权限判断会让行为不明确
public int Row
{
    get => Row;	// 始终返回自己
    set { if (IsMovable()) Row = value; }	// 这里的判断尽量不要写判断
}

6. 填充棋盘的思路

因为最后一排（row-1）已经到底了，不会再往下填充，所以我们从下往上，从倒数第二排开始，一行一行往上检查。

另外，第一排的情况要特殊考虑，因为它的上面 row=-1 处并没有生成棋子，所以我们需要创建一层 -1 层的棋子来等于第一排进行填充。

private bool FillStep()
{
    bool movedPiece = false;

    // 先从倒数第二排开始，一排一排往上逐个检查
    // 最后一排因为i不能往下移动了，所以不用管
    for (int row = dimensionY - 2; row >= 0; row--)
    {
        for (int col = 0; col < dimensionX; col++)
        {
            GamePiece piece = _pieces[col, row];
            if (_pieces[col, row].IsMovable())
            {
                // 检查它正下方的棋子
                GamePiece pieceBelow = _pieces[col, row + 1];
                // 检查pieceBelow的类型
                if (pieceBelow.EType == EPieceType.Empty)
                {
                    // 移动（探路 => 放置）
                    // 1）探路
                    piece.MovablePieceComponent.Move(row+1,col);
                    _pieces[col, row + 1] = piece;
                    // 2）放置
                    SpawnNewPiece(row, col, EPieceType.Empty);  // 原来位置的变成empty了
                    movedPiece = true;
                }
            }
        }
    }
