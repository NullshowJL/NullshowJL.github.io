2048

教程参考及素材来源：油管-Zigurous

源码： on Gitee

游戏简介

《2048》是一款由意大利开发者 Gabriele Cirulli 于 2014 年发布的数字益智滑块游戏，因其规则简单、上手容易但极具挑战性而迅速风靡全球。

核心玩法：

  • 游戏界面：一个 4×4 的方格棋盘。
  • 初始状态：棋盘上随机出现两个数字方块，通常是 2 或 4。
  • 操作方式：玩家通过上下左右滑动（或按键）控制所有方块朝指定方向移动。
  • 合并规则：
    • 相同数字的两个方块在碰撞时会合并为一个新方块，数值为两者之和（如 2 + 2 = 4，4 + 4 = 8，……）。
    • 每次滑动后，空位中会随机生成一个新的 2 或 4。
  • 目标：通过不断合并，最终得到一个 数值为 2048 的方块（达成后可继续玩以追求更高分数）。
  • 失败条件：当棋盘被填满且没有任何相邻相同数字可以合并时，游戏结束。

游戏/功能拆解

内容

画布/背景、网格、瓦片（数字、颜色）、游戏结束判定、UI（分数）

逻辑

  1. 棋盘上随机出现两个相同的数字方块
  2. 玩家上下左右滑动方块，同样的方块合并为下一个目标数字（2 - 4 - 8 -16…）
  3. 当棋盘被填满而没有任何相邻数字可以合并时，游戏结束
  4. 显示当前的分数

遇到的问题及解决方法

1. 导入的字体在 TextMesh Pro 中使用

先确保安装了 TextMesh Pro，然后选中字体右键点击 Create => TextMesh Pro => Font Asset。

2. Cell 的快速设置

使用 Horizontal Layout Group 和 Vertical Layout Group。但是需要注意的是要边看边设置，否则会出现奇怪的效果。同时在此项目中，设置好了需要把这两个组件删除，否则加上瓦片移动的脚本后也会出现奇怪的效果。

3. 用 ScriptableObject 来存储 Tile 的状态数据

一个类如果继承了 ScriptableObject，意味着它将不是“场景里的组件”，而是“可序列化的Unity资产（Asset）容器”，可以以 .asset 的形式存在，被多个对象/场景引用。此项目的应用场景是下面第二种场景。

使用场景：

  1. 全局配置/数值表/平衡参数
    • 角色属性、武器参数、技能数据、关卡配置、掉落表等。
    • 优点：改数据不改代码，能做版本管理和复用。
  2. “数据驱动”的资源定义（Data Asset）
    • 例如“道具定义”“Tile 定义”“敌人类型定义”这种“模板数据”。
    • 优点：可被多个 Prefab/场景共享引用，避免复制粘贴配置。
  3. 共享的引用与依赖注入风格（Project 级单例数据）
    • 比如把音量设置、输入映射、全局事件通道（Event Channel）做成 。 ScriptableObject
    • 优点：解耦对象之间的直接引用，跨场景更方便。
  4. 编辑器友好的内容制作管线
    • 设计师/策划在 Inspector 里填表、拖引用，程序只负责读取与执行。
    • 优点：内容迭代快。

4. 网格初始化的思路

按照游戏物体的层级由低到高：

Cell（不动的单个格子）=> Row（负责管理它下面的一排Cell）=> Grid（网格，负责管理它下面的四排Row以及Row下面的四个格子）=> Board（负责管理不动的网格+动的Tile）

另外，Tile是一个个可以在单个格子（Cell）上移动的瓦片。

//======== Cell.cs =========
public class Cell : MonoBehaviour
{
    public Vector2Int Coordinates { get; set; }
    public Tile Tile { get; set; }
    
    // 瓦片状态
    public bool Empty => Tile == null;
    public bool Occupied => Tile != null;
}

//========== Row.cs ==========
public class Row : MonoBehaviour
{
    public Cell[] Cells { get; private set; }   // 只能由这一个Row设置，而不希望由别的类来设置

    private void Awake()
    {
        Cells = GetComponentsInChildren<Cell>();
    }
}

//========== Grid.cs ===========
public class Grid : MonoBehaviour
{
    public Row[] Rows { get; private set; }
    public Cell[] Cells { get; private set; }
    
    public int Size => Cells.Length;
    public int Height => Rows.Length;
    public int Width => Size / Height;
    
    private void Awake()
    {
        Rows = GetComponentsInChildren<Row>();
        Cells = GetComponentsInChildren<Cell>();
    }
    
    // 初始化所有的格子
    private void Start()
    {
        for (int y = 0; y < Height; y++)
        {
            for (int x = 0; x < Width; x++)
            {
                Rows[y].Cells[x].Coordinates = new Vector2Int(x, y);
            }
        }
    }
}

5. 瓦片生成的实现思路

按照游戏物体的层级，简要来说，在这一阶段Board负责“生成什么”、Grid决定“放在哪里”、Tile执行“放进格子并确保和Cell的双向引用一致”。

具体来说：

  • Board
    • 控制游戏流程：什么时候生成新瓦片、生成几个、生成什么数字/状态
    • 维护一个 _tiles 列表（方便以后遍历、移动、合并、判断输赢等）
  • Grid
    • 管理所有 Cell（格子）
    • 提供工具方法：比如 GetRandomEmptyCell() 随机拿一个空格子
  • Cell
    • 单个格子的数据容器
    • Cell.Tile 指向“这个格子当前放着哪个 Tile”
  • Tile
    • 单个瓦片（UI + 数值）
    • Tile.Number/State 决定显示内容
    • Tile.Cell 记录“我现在在哪个 Cell 上”
    • Spawn(cell) 负责把自己放到某个格子，并把双方引用绑好

需要确保Tile和Cell之间的双向绑定：

  • 从 Cell 看：Cell.Tile 能告诉我格子里是什么瓦片
  • 从 Tile 看：Tile.Cell 能告诉我瓦片在哪个格子

这两者必须永远一致，否则：

  • Grid 以为某格子是占用的，但视觉上瓦片已经走了（或反过来）
  • 随机找空格子会找错
  • 移动/合并判断会乱套

所以生成/移动时就需要：

  1. 旧关系清掉（解绑）
  2. 新关系建立（绑定）

6. 瓦片移动的实现思路

整体思路：

按顺序遍历所有的格子 => 对每个有瓦片的格子执行 MoveTile => 在MoveTile 时先沿指定方向探路找最终落点/合并目标 => 最后用 MoveToCell 一次性更新Cell.Tell和Tell.Cell 的位置。

Tile.cs
  • MoveToCell / Spawn：
    • 清空/解绑旧格子
    • 绑定新格子
    • 让tile和cell的位置一致
Board.cs
  • Update里接受玩家输入
  • MoveTiles（全部瓦片的移动）
    • 单个瓦片移动 MoveTile
    • 棋盘上所有瓦片的移动顺序（避免同一片瓦片被处理两次）
  • MoveTile
    • 探路
      • 前方是null（出界）
      • 前方是empty（还能走，继续更新）
      • 前方是occupied（合并/被挡住停止）
    • 落子
      • 有newCell（tile.MoveToCell(newCell)）
Grid.cs (Helper - 提供cell的信息)
  • GetCell
  • GetAdjacentCell
